## 需求完整性 (Requirement Completeness)

- [x] CHK001 - US1 验收场景 #1 要求"首页与正文至少 80% 版式风格一致"，但未定义剩余 20% 不一致时的处置策略（降级提示 / 标记不一致 / 用户手动修正）？[Gap, Spec §US1-Scenario1] → **已修复**：补充"剩余不一致页面在风格契合度报告中标注差异维度，用户可选择'接受'或'重做该页'"
- [x] CHK002 - 是否为所有用户故事定义了"知识库为空"的完整降级路径？US1 有明确引导（FR-002），但 US6 素材检索在知识库为空时的行为未指定。[Completeness, Spec §US6] → **已修复**：US6 Acceptance Scenario #1 增加"素材库为空"的引导提示场景
- [x] CHK003 - 是否定义了可访问性（Accessibility）需求？当前 spec 未提及键盘导航、屏幕阅读器、色彩对比度等要求，而目标用户包含企业客户（合规审计）。[Gap] → **已修复**：新增 FR-038（浏览器兼容）和 FR-039（WCAG 2.1 AA 可访问性）
- [x] CHK004 - FR-027 定义了"生成任务结果保留 180 天"，但未指定用户订阅到期或账号休眠期间的保留行为是否变化。[Gap, Spec §FR-027] → **已修复**：明确"180 个自然日"，补充"订阅到期/休眠期间保留策略不变"
- [x] CHK005 - 是否定义了"风格归一"批量插入（一次插入多张复用页）时的需求？FR-034 仅描述单张页插入场景，批量时部分成功/部分失败的处置未覆盖。[Gap, Spec §FR-034] → **已修复**：FR-034 补充"批量插入时逐页独立执行归一，汇总结果通知用户"

## 需求清晰度 (Requirement Clarity)

- [x] CHK006 - "风格"一词在 spec 中至少出现三种语境（样本风格、草稿整体风格、风格契合度），是否在某处给出了统一的操作性定义？[Ambiguity] → **已修复**：新增"术语定义"章节，定义"风格"三维度（版式结构/配色分布/字体族）、"同类型修改"、"草稿整体风格"
- [x] CHK007 - US6 验收场景 #2 中"≤1 张页出现明显样式冲突"的"明显"是否可量化？当前依赖主观判断，无法客观验收。[Measurability, Spec §US6-Scenario2] → **已修复**：改为"风格三维度中任意两维度命中 ≥80% 的页面占比 ≥90%"
- [x] CHK008 - SC-007 要求"前 5 次生成中重复 ≥3 次的同类型修改"，"同类型修改"的判定标准是什么？（相同页面位置 / 相同操作类型 / 相同参数值）[Clarity, Spec §SC-007] → **已修复**：SC-007 引用"术语定义"中的"同类型修改"定义
- [x] CHK009 - SC-002 的"每月 1 次对 5% 抽样做 3 名产品经理盲评"——抽样方法、盲评流程、评分标准（5 分制各分值含义）是否需要在需求中定义？[Clarity, Spec §SC-002] → **已修复**：SC-002 补充完整盲评流程（抽样→评分人员→评分标准→签发要求）
- [x] CHK010 - FR-034 中"草稿整体风格"的操作性定义是什么？由"占多数的页风格聚类"决定，但聚类算法/阈值未指定——不同实现可能产出不同"整体风格"。[Ambiguity, Spec §FR-034] → **已修复**：在"术语定义"中定义"草稿整体风格"为"≥60% 页面的风格聚类结果"
- [x] CHK011 - SC-015 中"首条点开率 ≥60%"——"点开"的定义是什么？（缩略图点击 / 详情面板打开 / 插入草稿）[Clarity, Spec §SC-015] → **已修复**：SC-015 补充"点开 = 用户点击缩略图后进入素材详情面板"

## 需求一致性 (Requirement Consistency)

- [x] CHK012 - FR-037 要求"不污染原历史样本"，但 FR-034 的风格归一需要读取原样本的配色/字体/版式参数——两者是否冲突？若原样本被删除，归一参数从何获取？[Conflict, Spec §FR-034 vs §FR-037] → **已修复**：FR-034 明确"归一仅作用于派生副本，不修改原素材"；FR-037 补充"归一读取原样本参数用于计算，但仅写入派生副本"
- [x] CHK013 - SC-005 要求"样本删除后 1 小时内不可恢复"，FR-019 要求"删除全部数据后 24 小时内完成清除"——两者对"删除"的时效要求不一致，是否为不同场景（单样本 vs 全量）的有意区分？[Consistency, Spec §SC-005 vs §FR-019] → **已修复**：SC-005 补充说明"此为单样本删除的即时性要求，区别于 FR-019 全量清除的 24 小时窗口"
- [x] CHK014 - FR-009 要求"删除样本时三类数据同步清除"，但 US6 追加说明中允许 SlideAsset 作为"孤儿素材"保留——FR-009 的"同步清除"是否适用于 SlideAsset？[Conflict, Spec §FR-009 vs §US6 Clarification] → **已修复**：FR-009 补充"但由该样本派生的 SlideAsset 在被草稿引用时 MUST 保留为孤儿素材"
- [x] CHK015 - FR-029 规定"排队超过 5 分钟自动取消"，FR-025 规定"计划性维护提前 24 小时公告"——维护窗口期间排队任务是正常排队还是直接拒绝？两者未衔接。[Gap, Spec §FR-029 vs §FR-025] → **已修复**：FR-025 和 FR-029 均补充"维护窗口期间排队任务正常保留不取消，维护结束后按原队列顺序恢复"

## 验收标准质量 (Acceptance Criteria Quality)

- [x] CHK016 - SC-016 要求"10 人盲评主观一致度 ≥4 分"——评分人员来源（内部 / 外部）、评分维度（仅色板+字体 / 含版式）、评分标准文档是否需要在需求中明确？[Measurability, Spec §SC-016] → **已修复**：SC-016 补充完整盲评流程（抽样频率→评分人员→评分标准→签发要求）
- [x] CHK017 - SC-015 中"相关度 ≥0.6"——相关度评分的计算公式在 Clarification 中定义了权重（BM25 0.4 + 视觉类型 0.2 + 嵌入 0.4），但该公式是否应纳入 Success Criteria 作为验收基准？[Traceability, Spec §SC-015 vs §Clarification] → **已修复**：SC-015 直接内嵌相关度公式
- [x] CHK018 - SC-009 要求"重做阶段耗时降低 ≥60%"——基准是"完整重做"的耗时，但"完整重做"本身无 SLA（仅 SC-001 有 5 分钟总耗时），基准不固定则 60% 无法验证。[Measurability, Spec §SC-009] → **未修复**：此为度量设计问题，需在 plan.md 中定义基准测量方法（超出 spec 范围）

## 场景覆盖 (Scenario Coverage)

- [x] CHK019 - 是否定义了草稿导出失败（中途断网 / 服务端异常）的需求？FR-036 定义了成功路径的溯源元数据写入，但失败时的重试/回滚/用户通知未覆盖。[Coverage, Exception Flow, Spec §FR-036] → **已修复**：FR-036 补充"导出异常时 30 秒内返回失败原因（RFC 7807），保留部分文件，记录安全事件"
- [x] CHK020 - 是否定义了"风格归一"在不同页之间部分成功、部分失败时的批量处置需求？当前 FR-034 仅描述单张页失败时"保留原样+打标签"，批量场景未覆盖。[Coverage, Spec §FR-034] → **已修复**：FR-034 补充批量归一逻辑（与 CHK005 同一修复）
- [x] CHK021 - 是否定义了用户在编辑草稿时锁到期（30 分钟无操作）的自动保存与退出行为？FR-035 定义了自动保存，但锁到期后未保存的变更如何处理未指定。[Coverage, Edge Case, Spec §FR-035] → **已修复**：新增 Edge Case "锁到期未保存"
- [x] CHK022 - 是否定义了"一键删除全部数据"执行中用户又发起新生成任务的竞态需求？FR-019 的 24 小时清除窗口与 FR-001 的实时生成可能冲突。[Coverage, Concurrent Flow, Spec §FR-019 vs §FR-001] → **已修复**：新增 Edge Case "删除与生成竞态"
